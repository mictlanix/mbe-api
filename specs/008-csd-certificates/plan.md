# Implementation Plan: Digital Seal Certificates

**Branch**: `010-open-issues-fixes`, `012-csd-validity-local-time` | **Date**: 2026-07-21 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/008-csd-certificates/spec.md`

## Summary

Expose `taxpayer_certificate` ‚Äî modelled but previously unreachable ‚Äî as metadata-only reads
plus a validated upload, gated by `SystemObject.TAXPAYERS` (24).

- **Reads** return `taxpayer_certificate_id`, `taxpayer`, `valid_from`, `valid_to`, `status`.
  The `certificate_data` / `key_data` binaries and the raw `key_password` are excluded by
  `load_only` from the query itself, not merely from the response schema.
- **Upload** takes a multipart CSD pair and validates it in `app/services/csd_service.py`
  before storing: the password must open the DER PKCS#8 key, the key's public numbers must
  match the certificate's, and the RFC in the certificate subject must match the issuer.
  Certificate number and validity are read from the X.509, not the form.
- **Validity is stored as `America/Mexico_City` local time**, matching every row the legacy
  system wrote (PR #106).

## Technical Context

**Language/Version**: Python 3.12+

**Primary Dependencies**: FastAPI, SQLAlchemy 2.0 async, Pydantic v2, plus `cryptography`
promoted from a transitive dependency of `python-jose` to a direct one, and `tzdata`

**Storage**: MariaDB via aiomysql. No schema change ‚Äî `taxpayer_certificate` already exists
with its three `LargeBinary` columns.

**Testing**: pytest + pytest-asyncio + httpx `ASGITransport`, with CSD pairs generated by
`cryptography` in-test, plus verification against the 17 real certificates in `mbe_demo`

**Target Platform**: Linux server (FastAPI/ASGI)

**Project Type**: Web service ‚Äî one endpoint module, two services, one schema addition

**Performance Goals**: Certificate parsing is CPU-bound and runs off the event loop via
`asyncio.to_thread`, matching the existing image-upload service. Reads must never load the
binaries: a list of 20 certificates would otherwise pull megabytes of key material.

**Constraints**:
- Secret material must not be *selected*, not merely not serialized (spec FR-005).
- `key_password` stays unencrypted at rest ‚Äî an explicit product decision, out of scope here.
- Validity must match the legacy convention; storing UTC would put new rows 6 hours off.
- Timezone data must not depend on the host carrying a system tz database.

**Scale/Scope**: 1 new endpoint module, 2 new services, 1 schema addition, 2 dependencies
promoted/added; 3 test files.

| V. Reuse Over Rebuild | ‚öÝÔ∏è | Two new services. Justified in Complexity Tracking. |
| VI. Async-First | ‚úÖ | Handlers and services are `async def`; the CPU-bound parse is dispatched to a thread rather than blocking the loop. |

against real DER material generated in-test, not mocked ‚Äî a mocked crypto library would prove
nothing about the assumptions being made.

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|-------------------------------------|
| Two services rather than one (Principle V) | `csd_service` is pure cryptography with no session, dispatched to a thread; `taxpayer_certificate_service` is persistence with no cryptography. | Folding the parsing into the persistence service would put CPU-bound work behind an `AsyncSession` signature and make the parser untestable without a database. The split is what allows `test_csd_service.py` to exercise real DER material with no session at all. |
| `cryptography` as a direct dependency | Required to parse X.509 and PKCS#8. | Already present transitively via `python-jose`; declaring it direct records the actual dependency rather than relying on another package's transitive graph. |
| `tzdata` dependency | `zoneinfo` falls back to it when the host has no system tz database ‚Äî the normal case in slim containers. | Hard-coding a fixed offset was rejected: correct only since October 2022, and silently wrong for any historical date. |
